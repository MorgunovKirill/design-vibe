# Spec for Supabase Authentication and Dashboard

figma_component (if used): N/A

## Summary

Внедрение полноценной аутентификации через Supabase Auth и создание динамического дашборда с хранением задач в базе данных Supabase. Пользователи смогут регистрироваться, входить в систему (через email/пароль и OAuth), а их задачи будут сохраняться и загружаться из Supabase.

## Functional Requirements

### Аутентификация

1. **Регистрация (Sign Up)**
   - Форма email + пароль подключается к `supabase.auth.signUp()`
   - После успешной регистрации — редирект на `/dashboard`
   - Отображение ошибок валидации (email занят, слабый пароль и т.д.)

2. **Вход (Sign In)**
   - Форма email + пароль подключается к `supabase.auth.signInWithPassword()`
   - После успешного входа — редирект на `/dashboard`
   - OAuth: кнопки Google и GitHub подключаются к `supabase.auth.signInWithOAuth()`

3. **Выход (Sign Out)**
   - Кнопка в дашборде вызывает `supabase.auth.signOut()`
   - Редирект на `/sign-in` после выхода

4. **Auth Guard**
   - `src/app/(platform)/layout.tsx` проверяет сессию через `supabase.auth.getUser()`
   - Неаутентифицированные пользователи перенаправляются на `/sign-in`

5. **Синхронизация состояния сессии**
   - Использование `supabase.auth.onAuthStateChange()` для обновления UI при смене сессии

### База данных

6. **Таблица `tasks`**
   - Поля: `id`, `user_id` (FK → `auth.users`), `title`, `status` (`'To Start' | 'In Progress' | 'Done'`), `category` (`'Design' | 'Front-end'`), `due_date`, `created_at`, `updated_at`
   - Row Level Security (RLS): пользователь видит и редактирует только свои задачи

7. **CRUD операции**
   - Загрузка задач текущего пользователя при открытии дашборда
   - Создание новой задачи через форму/модальное окно
   - Обновление статуса задачи (drag-and-drop или через меню)
   - Удаление задачи

### Дашборд

8. **Динамические данные**
   - Замена `MOCK_TASKS` на данные из Supabase
   - Загрузочное состояние (skeleton или spinner) пока данные загружаются
   - Пустое состояние при отсутствии задач

## Possible Edge Cases

- Пользователь переходит на `/dashboard` без сессии → редирект на `/sign-in`
- Истечение сессии во время работы → повторная авторизация
- Ошибка сети при загрузке задач → отображение сообщения об ошибке
- OAuth провайдер недоступен → graceful fallback с сообщением
- Попытка регистрации с уже существующим email
- Одновременное открытие в нескольких вкладках (синхронизация через `onAuthStateChange`)

## Acceptance Criteria

- [ ] Пользователь может зарегистрироваться через email/пароль
- [ ] Пользователь может войти через email/пароль
- [ ] Пользователь может войти через Google и GitHub (OAuth)
- [ ] Неавторизованный пользователь перенаправляется с `/dashboard` на `/sign-in`
- [ ] После входа пользователь видит свои задачи из Supabase
- [ ] Пользователь может создать новую задачу
- [ ] Пользователь может изменить статус задачи
- [ ] Пользователь может удалить задачу
- [ ] Каждый пользователь видит только свои задачи (RLS)
- [ ] Кнопка выхода корректно завершает сессию и перенаправляет на `/sign-in`

## Open Questions

- Нужна ли верификация email при регистрации, или можно сразу пускать на дашборд?
- Использовать Supabase SSR (`@supabase/ssr`) или клиентский SDK — важно для Next.js 15 App Router
- Нужен ли drag-and-drop для перетаскивания карточек между колонками, или достаточно смены статуса через меню?
- Хранить `due_date` как обязательное поле или опциональное?
- Нужны ли уведомления (toast) при успешных операциях (создание/удаление задачи)?
